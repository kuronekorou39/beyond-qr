画面-カメラ間 光学データ伝送 — 関連研究・先行事例まとめ

beyond-qr プロジェクト(アニメーション QR + fountain code による PC→スマホ画像転送)の 位置づけを確認するための先行事例サーベイ。2026-07-02 時点の調査。

スループット比較(実測値)

システム方式実測スループット備考
beyond-qr(本プロジェクト) 標準 QR + fountain code ~6.7 KB/s(53 kbps) 200KB / 30秒。MLKit デコード、Flutter 実機
TXQR 標準 QR + LT fountain code 最良 ~9 KB/s / 実用 1〜2 KB/s iPhone、近距離・静止の最良条件
InFrame++(学術) 可視動画への埋め込み 50.4 kbps 動画に不可視で埋め込むという別制約下
MAMBA(学術) 適応型バーコード(双方向) 69.1 kbps スマホ画面 ↔ スマホカメラ
RDCode(MobiCom 2014) 専用動的バーコード COBRA 比 約2倍 3階層(packet-frame-block)の独自誤り訂正

読み方: 「標準 QR + 汎用デコーダ」という同一条件では、実用システムは概ね 50〜70 kbps 帯に集中しており、本プロジェクトはそのゾーンの中央にいる。 桁が変わる(300 kbps〜1 Mbps 級)のは QR を捨てて専用コード + 専用デコーダに移行した システムのみ。

各ソースの詳細

TXQR — Animated QR data transfer

OSSGo最も近い先行例 Ivan Daniluk (divan)、2018–2019
  • 動機: 政府検閲などでネットワークが使えない状況を想定し、アニメーション QR だけでデバイス間データ転送を実現する。
  • プロトコル: 各フレームに NUM/TOTAL| プリフィックス + Base64 ペイロード。当初は単純な繰り返し(ループ)方式。
  • 課題と解決: 繰り返し方式では 1 フレーム見逃すと 1 周待ちになる。これを LT 符号(fountain code、Google の gofountain を使用)で解決し、「どのフレームから受けても K 個+α 集まれば復元できる」構造に変更。Soliton 分布により XOR するブロック数のバランスを取るのが鍵。
  • 実測: 最良で 約 9 KB/s(11 FPS・チャンク 850 バイト・EC レベル M)。fountain code 導入後は約 13KB を 501ms で転送する記録も。ただし実用的には 1〜2 KB/s、安定動作は 6〜7 FPS・550〜900 バイトチャンク。
  • 本プロジェクトとの関係: 「QR + fountain code + 動画ループ」という構成がほぼ同一。チャンクサイズ・FPS・EC レベルのチューニングが支配的という結論も一致する。
出典: Animated QR data transfer with Gomobile and Gopherjs / Fountain codes and animated QR / GitHub: divan/txqr

libcimbar — Color Icon Matrix Barcodes

OSSC++/WASM最速クラス sz3
  • 概要: QR とは非互換の独自 2D コード「cimbar」。エアギャップ転送(モニター → スマホカメラのみ)を目的とし、852 kbps(~106 KB/s) を実測。4,689,084 バイト(zstd 圧縮後)を 44 秒で転送。
  • エンコード方式: 8×8 ピクセルのタイルシンボル(形状で 4 bit)+ 4 色(色で 2 bit)= 6 bit/タイル。1 画像あたり理論 9,300 バイト、Reed-Solomon(155 バイト中 30 バイト訂正)適用後の実効 7,500 バイト。
  • fountain code: wirehair を使用。フレームの欠落・順序入れ替わり・部分破損に耐える。zstd 圧縮込みで最大 33MB のファイルに対応。
  • 実測環境: 送信は cimbar.org(WASM、ブラウザで動作)、受信は Android アプリ(Snapdragon 625、自動露出/AF)。OpenCV ベースの専用デコーダ。
  • 本プロジェクトへの示唆: 「4 色 + タイル形状の併用」で色キャリブレーション問題を回避しており、Phase 0 のカラー方式で直面したクリッピング/較正問題への一つの答えになっている。10 倍級の高速化を目指す場合の参照実装。
出典: GitHub: sz3/libcimbar / PERFORMANCE.md

COBRA — Color Barcode Streaming for Smartphone Systems

学術MobiSys 2012 Tian Hao, Ruogu Zhou, Guoliang Xing
  • 概要: 画面-カメラ間ストリーミング(screen-camera link)研究の草分け。スマホの小型画面・低速カメラ向けに最適化した独自 2D カラーバーコードを設計し、スマホ間でリアルタイムに動くコードをストリーミングする。
  • 特徴: 手持ちによるブレ(モーションブラー)に対応するため、コードブロックのサイズと配置を動的に適応させる。可視光の指向性・短距離性により、近距離データ交換でのプライバシー保護が利点であることを明示的に主張している(本プロジェクトの「通信を使わない=傍受されにくい」という動機と同じ)。
  • 位置づけ: 後続研究(RDCode 等)のベースライン。RDCode は COBRA 比で約 2 倍のスループットを報告している。
出典: ACM DL: COBRA (論文本文は有料のため、スループットの詳細数値は本ページでは省略)

RDCode — Enhancing Reliability to Boost the Throughput over Screen-Camera Links

学術MobiCom 2014
  • 概要: 画面-カメラ間リンクの信頼性を上げることでスループットを引き上げる、という本プロジェクトと同じ問題意識の研究。
  • 方式: packet-frame-block の 3 階層構造を持つ動的バーコードを設計し、階層ごとに異なる誤り訂正方式を適用(ブロック内訂正 / フレーム内のブロック消失訂正 / フレーム消失訂正)。フレーム消失への対処という点で fountain code と同じ役割を階層 EC で実現している。
  • 結果: COBRA 比で約 2 倍の伝送レート。近距離・大画面条件では 300 kbps 超級の報告もある。
出典: ACM DL: RDCode (PDF)

MAMBA / InFrame++ — その他の学術システム

学術
  • MAMBA: チャネル状態に応じてコード密度を適応させる双方向型のカメラ-ディスプレイ通信。スマホ実機で 69.1 kbps を報告。「実機スマホ + 動的バーコード」の実用上限の目安になる数字。
  • InFrame++: 人間には普通の動画に見える映像の中に不可視データを埋め込む方式(視認性と伝送の両立という別の制約付き)で 50.4 kbps。制約があっても 50 kbps 出るという意味で、このレンジが「スマホカメラ受信の現実的な帯域」であることの傍証。
出典: MAMBA (ResearchGate) / InFrame++ (ResearchGate)

結論 — 本プロジェクトの位置づけ

  • 200KB / 30秒(53 kbps)は同条件の先行例と比べて遅くない。 標準 QR + 汎用デコーダ(MLKit)の構成では TXQR の最良値(~9 KB/s)に近く、学術系スマホ実機システム(50〜70 kbps)の中央値レベル。
  • 技術的な合法性・正当性にも問題はない。 エアギャップ転送は COBRA が論文でプライバシー上の利点として主張し、暗号資産ハードウェアウォレット(BC-UR 規格: Keystone、AirGap Vault、SeedSigner 等)が商用実運用している確立された用途。ただし組織内で DLP を回避する使い方は利用者側の規程違反になり得る。
  • 10 倍級の高速化を目指すなら libcimbar 路線(限定色 + タイルシンボル + 専用 RS 訂正 + 自前デコーダ)への設計変更が必要で、MLKit を捨てる規模の話になる。現要件(200KB/1分)には現状で 2 倍のマージンがあるため不要。
