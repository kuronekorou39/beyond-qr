# vcode フレームフォーマット仕様 (v1)

> v0→v1 (2026-07-10): ヘッダ/ブロックの CRC を 16→32 bit 化し、fountain ペイロード先頭に
> エンドツーエンド CRC-32 を追加。サブセルオフセットリトライ (25 通り) x 長時間受信では
> CRC-16 の偽陽性 (1/65536) が現実に発生し、ゴミパケットが RaptorQ を汚染して復元結果
> 全体が壊れるため (800KB/300s の実機受信で顕在化)。v0 とは非互換 (version 検証で棄却)。

動画ネイティブな独自 2D コード。QR が印刷物向けに払っているコスト(ファインダー/アライメントパターン、全か無かの RS デコード)を捨て、画面→カメラのストリーミング専用に再設計したもの。実装は `vcode/` クレート。

## 設計原則

1. **部分回収**: フレームは独立したブロックの格子。各ブロックが RaptorQ パケット 1 個 + CRC-32 を持ち、部分的に壊れたフレームからも読めたブロックだけ回収する。QR の「1 フレーム = 成功 or 失敗」を連続量に変えるのが最大の狙い。
2. **フレーム内 EC なし**: フレーム/ブロックの欠落は上位の fountain (RaptorQ) が吸収する。QR の EC=M(約 25% 冗長)に対し、vcode の冗長はブロックあたり CRC 4 バイト(8%)のみ。
3. **どのフレームからでも受信開始可能**: OTI(12 バイト)を全フレームヘッダに載せる。プロトコル外での OTI 共有が不要になる。
4. **将来拡張**: ヘッダに `bits_per_cell` を持ち、輝度 4 値(2bit/セル)への拡張に備える。v0 は 1bit(白黒)。

## フレーム構造(デフォルトレイアウト: 100×92 セル)

```
+------+----------------------------------+------+
| TL   |  行 0: 擬似ランダムタイミング行      | TR   |  行 0..6
| 6x6  |  行 1..6: ヘッダ (24B、CRC込 × 2 コピー)| 6x6  |
+------+----------------------------------+------+
|                                                |
|   データ領域: 20×20 セルのブロック × 5×4 = 20 個  |  行 6..86
|   ブロック = 46B ペイロード + CRC-32 = 50B       |
|                                                |
+------+----------------------------------+------+
| BL   |  擬似ランダム較正パターン             | BR   |  行 86..92
| 6x6  |                                  | 6x6  |
+------+----------------------------------+------+
```

- **コーナーマーカー(6×6)**: 外周 1 セルは全コーナー共通で黒(検出用)。内部 4×4 はコーナーごとに異なるパターン(TL=塗りつぶし / TR=白抜き / BL=中央 2×2 黒 / BR=市松)で、回転・鏡像の判定に使う。
- **較正パターン(行 0 + 下ストリップ)**: 座標ハッシュによる擬似ランダム既知パターン。市松等の周期パターンは 2 セルずれても一致する「位相トラップ」を作るため使わない(実機デバッグで判明)。上下両側に置くことで水平スケール誤差を拘束する。多値化時は既知階調ストリップを兼ねる予定。
- **ヘッダ(24 バイト)**: `magic(0xB9) | version | bits_per_cell | block | grid_w | grid_h | frame_seq(u16 LE) | OTI(12B) | CRC-32(BE)`。行 1..6 の 440 セルに 2 コピー。version 不一致は棄却。デコーダは CRC が通る最初のコピーを採用。
- **ブロック**: セルを行優先で MSB-first にビット詰め。1=黒。ペイロード 46 バイト = シリアライズ済み RaptorQ パケット(4B payload ID + packet_size 42B)がそのまま入る。

## 容量比較

| | QR v18 / EC=M | vcode v0 (100×92) |
|---|---|---|
| セル(モジュール)数 | 89×89 = 7,921 | 100×92 = 9,200 |
| 実効ペイロード/フレーム | ~720 バイト | 840 バイト (42B × 20 パケット) |
| 破損時の挙動 | 全損 | 読めたブロックだけ回収 |
| 機能パターン+冗長の面積比 | ~35%(機能パターン+EC) | ~19%(ストリップ+コーナー+CRC) |

セル数比 1.16 倍で実効 1.17 倍。密度は上げていないので頑健性は犠牲にしていない。実効レートの本命は部分回収とトラッキング(次段)。

## デコード経路

- `decode_frame()`: 理想チャネル(整数スケール・歪みなし)用。ベンチ・テストで使用。
- `scan::scan_frame()`: 実カメラ画像用。UI のガイド枠を初期値として、
  1. **粗探索**: 各コーナー独立に ±32px を 2px 刻みで全数探索(評価はそのコーナーのマーカーセルのみ。全数なのでマーカー自己相似の局所最適に捕まらない)
  2. **微調整**: 全既知セル(コーナー + 擬似ランダム較正)の一致数を最大化する座標降下(2/1/0.5px)
  3. **Otsu 二値化**(argmax 平坦区間の中央を排他的閾値として返す実装。オフバイワンに注意)
  4. **サブセルオフセットリトライ**: ヘッダ/各ブロックを ±0.5 セルの 5×5 オフセットで再読し、CRC が通る位置を採用(残留ずれの局所吸収)

  位置合わせの評価は常にコード内部の既知セルのみで行うため、周辺のクラッタ(ブラウザ UI、テキスト等)の影響を受けない。

## 輝度 4 値 (bits_per_cell = 2)

データブロックのみ 4 階調 (0/85/170/255、2bit/セル、MSB-first のビットペア → レベル)。
構造セル (コーナー/ヘッダ/較正) は白黒のまま — 位置合わせとヘッダ読みの頑健性を維持する。
受信は上下ストリップの既知白黒セルから列ごとに黒/白レベルを推定し、行方向は上下の
線形補間で正規化してから [1/6, 1/2, 5/6] 境界で量子化する (照明勾配・ガンマの局所較正)。

注意: raptorq は packet_size をシンボル整列 (8 の倍数等) に丸めるため、実パケット長は
OTI の symbol_size から求める。ブロックペイロードの余りはゼロパディング
(送信 = `VcodeTx`、受信の切り出し = `fountain::oti_symbol_size`)。

## エンドツーエンド検証 (v1)

送信ペイロードは fountain 符号化の前に `wrap_payload()` で先頭 4 バイトに CRC-32 が付く。
受信は復元完了時に `unwrap_payload()` で検証し、不一致なら復元結果を破棄して
デコーダを作り直し受信を続行する (RaptorQ は一度汚染されると復旧できないため)。
ブロック CRC-32 で偽陽性はほぼ消えるが、壊れたファイルを黙って保存しないための最終防衛線。

## 実機実測 (2026-07-02, Pixel 9a + PC モニタ, 200KB, release)

| 構成 | 実効スループット | QR 版比 |
|---|---|---|
| QR v18/EC=M + MLKit (トラック A) | 6.7 KB/s | 1.0x |
| vcode v0 (5x4, 1bit, 720p, 追従なし) | 7.85 KB/s | 1.2x |
| + フレーム間トラッキング | 14.5 KB/s | 2.2x |
| + 高密度 7x6 + 1080p | 19.6 KB/s | 2.9x |
| + 輝度 4 値 (2bit) | **26.8 KB/s** | **4.0x** |

送信 fps の最適点はカメラ 30fps に対し、1bit=20fps / 2bit=15fps (クリーンキャプチャ保証の重みが増す)。

次段: ガイドなし自動検出、双方向適応 (fps/密度/階調の自動選択)、カメラ 60fps 化。
