# vcode v0 フレームフォーマット仕様

動画ネイティブな独自 2D コード。QR が印刷物向けに払っているコスト(ファインダー/アライメントパターン、全か無かの RS デコード)を捨て、画面→カメラのストリーミング専用に再設計したもの。実装は `vcode/` クレート。

## 設計原則

1. **部分回収**: フレームは独立したブロックの格子。各ブロックが RaptorQ パケット 1 個 + CRC-16 を持ち、部分的に壊れたフレームからも読めたブロックだけ回収する。QR の「1 フレーム = 成功 or 失敗」を連続量に変えるのが最大の狙い。
2. **フレーム内 EC なし**: フレーム/ブロックの欠落は上位の fountain (RaptorQ) が吸収する。QR の EC=M(約 25% 冗長)に対し、vcode の冗長はブロックあたり CRC 2 バイト(4%)のみ。
3. **どのフレームからでも受信開始可能**: OTI(12 バイト)を全フレームヘッダに載せる。プロトコル外での OTI 共有が不要になる。
4. **将来拡張**: ヘッダに `bits_per_cell` を持ち、輝度 4 値(2bit/セル)への拡張に備える。v0 は 1bit(白黒)。

## フレーム構造(デフォルトレイアウト: 100×92 セル)

```
+------+----------------------------------+------+
| TL   |  ヘッダ (22B+CRC を 3 コピー)       | TR   |  行 0..6
| 6x6  |  528 セル = 176bit × 3            | 6x6  |
+------+----------------------------------+------+
|                                                |
|   データ領域: 20×20 セルのブロック × 5×4 = 20 個  |  行 6..86
|   ブロック = 48B ペイロード + CRC-16 = 50B       |
|                                                |
+------+----------------------------------+------+
| BL   |  市松の較正/タイミングパターン         | BR   |  行 86..92
| 6x6  |                                  | 6x6  |
+------+----------------------------------+------+
```

- **コーナーマーカー(6×6)**: 外周 1 セルは全コーナー共通で黒(検出用)。内部 4×4 はコーナーごとに異なるパターン(TL=塗りつぶし / TR=白抜き / BL=中央 2×2 黒 / BR=市松)で、回転・鏡像の判定に使う。
- **ヘッダ(22 バイト)**: `magic(0xB9) | version | bits_per_cell | block | grid_w | grid_h | frame_seq(u16 LE) | OTI(12B) | CRC-16(BE)`。上ストリップに入るだけコピーを繰り返す(100 セル幅なら丁度 3 コピー)。デコーダは CRC が通る最初のコピーを採用。
- **ブロック**: セルを行優先で MSB-first にビット詰め。1=黒。ペイロード 48 バイト = シリアライズ済み RaptorQ パケット(4B payload ID + packet_size 44B)がそのまま入る。
- **下ストリップ**: (row+col) 偶奇の市松。将来のグリッド位相合わせ・多値化時の較正ストリップ(既知階調)置き場。

## 容量比較

| | QR v18 / EC=M | vcode v0 (100×92) |
|---|---|---|
| セル(モジュール)数 | 89×89 = 7,921 | 100×92 = 9,200 |
| 実効ペイロード/フレーム | ~720 バイト | 880 バイト (44B × 20 パケット) |
| 破損時の挙動 | 全損 | 読めたブロックだけ回収 |
| 機能パターン+冗長の面積比 | ~35%(機能パターン+EC) | ~17%(ストリップ+コーナー+CRC) |

セル数比 1.16 倍で実効 1.22 倍。密度は上げていないので頑健性は犠牲にしていない。実効レートの本命は部分回収とトラッキング(次段)。

## v0 デコーダの前提と次段

`decode_frame()` は理想チャネル(整数スケール・歪みなし・二値)を仮定する。実カメラ画像に対しては、次段で以下を別モジュールとして実装する:

1. コーナーマーカー検出 → ホモグラフィ推定 → セル中心サンプリング
2. 動画での四隅トラッキング(毎フレームのフルサーチ回避)+ ホモグラフィの時間平滑化
3. 適応二値化(照明勾配対応)、その後に輝度 4 値化(2bit/セル)

## テスト

- `vcode/src/lib.rs`: CRC 標準ベクタ、ヘッダ/フレームのラウンドトリップ(scale 1/3)、フィラーブロックの棄却、部分破損時の無傷ブロック回収、コーナー破損検出
- `vcode/tests/fountain_roundtrip.rs`: fountain (RaptorQ) との結合。無損失 E2E と、各ブロック 30% 破損チャネルでの E2E 復元(リペア 60% 追加)
